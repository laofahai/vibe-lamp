# Vibe Lamp — AI 编码状态实体氛围灯 · 设计文档

- 日期：2026-06-13
- 状态：设计已确认，待进入实现计划

## 1. 背景与目标

Vibe Island 是一款 macOS 刘海/菜单栏面板，把 Claude Code、Codex 等 AI 编码 agent 的实时状态显示在刘海里（看状态、批准权限、跳回终端）。

**Vibe Lamp 是它的"实体灯版"**：用 ESP32 驱动一圈 LED，把当前编码会话的状态搬到桌面上一盏灯上——余光一瞥就知道 agent 在干活、干完了、还是卡住要你介入，不用盯屏幕。

### 核心价值
- **环境感知**：状态做成桌面氛围光，不抢注意力，但需要时一眼可见
- **不说谎**：灯永远反映真实状态；断线时显示"失联"而非冻结在旧颜色误导你
- **多 agent**：同时支持 Claude Code 和 Codex（架构上可扩展更多）

## 2. 整体架构（三层）

```
┌─────────────────────────────────────────────┐
│  你的 Mac                                      │
│  Claude Code ──hooks──┐                        │
│  Codex ───────hooks───┼──→ 转译器(常驻后台)      │
│  (以后更多 agent) ─────┘       合并所有会话状态    │
│                               决定该显示什么      │
│                                   │             │
│                          WiFi HTTP 推送 + 心跳   │
└───────────────────────────────────│────────────┘
                                    │ (WiFi断→v1.1切BLE兜底)
                                    ▼
                         ┌────────────────────┐
                         │ ESP32 (傻瓜显示端)   │
                         │ 收指令→驱动 LED      │
                         │ 看门狗→失联检测       │
                         └─────────│──────────┘
                                   ▼
                          WS2812 灯环（出色彩+动效）
```

### 设计原则
- **职责分离**：业务逻辑（状态合并、agent 差异）全在 Mac 端的转译器里；ESP32 只是傻瓜显示端，只懂"亮什么"。
- **ESP32 保持哑终端**：不解析事件、不懂 agent、不存业务状态。加新 agent 只改转译器，固件一行不动。
- **可靠性优先**：断线、进程崩溃、设备重启都要优雅处理，灯绝不冻在误导性的旧状态。

## 3. 组件设计

### 3.1 采集层（各 agent 的钩子）

各 agent 用自带的钩子机制，在状态变化时调用本地转译器的轻量入口。

**Claude Code**（`~/.claude/settings.json` 的 hooks）：
| 事件 | 语义 |
|---|---|
| `SessionStart` | 会话开始 |
| `UserPromptSubmit` | 用户提交指令 → 开始干活 |
| `PreToolUse` | 即将调用工具（payload 带工具名）→ 干活中 + 心跳 |
| `PostToolUse` | 工具完成（payload 带工具名、是否报错）→ 出错检测 |
| `Notification` | 需要权限/注意 → 要你介入 |
| `Stop` | 主回合结束 → 完成 |
| `SessionEnd` | 会话结束 → 空闲 |

**Codex**（`~/.codex/config.toml` 的 hooks + notify）：
| 事件 | 语义 |
|---|---|
| `SessionStart` / `UserPromptSubmit` / `Stop` | 默认安装；开始/干活/完成 |
| `PreToolUse` / `PostToolUse` | 需手动配置；干活细节/出错 |
| notify `agent-turn-complete` | 完成 |
| notify `approval-requested` | 要你介入 |

**安装**：一次性安装脚本自动把钩子条目写入两个配置文件（幂等，可重复运行；保留用户已有配置）。

### 3.2 转译/聚合层（Mac 端常驻守护进程）

整个系统的"大脑"。职责：
1. **接收事件**：暴露一个本地入口（如监听 Unix socket / 本地 HTTP），各 agent 钩子把事件喂进来。钩子那一行保持极简（一次性命令，触发即退出）。
2. **维护会话表**：记录每个活动会话（agent 类型、session id、当前状态、最后活动时间、当前工具）。
3. **合并状态**：把所有会话状态算成"该给灯显示什么"。
4. **推送显示**：经 WiFi HTTP 把显示指令推给 ESP32，并定期发心跳。
5. **健壮性**：推送失败时静默重试 + 缓存最后状态，不崩；会话超时无活动则兜底回空闲。

**合并优先级**（单灯/整体氛围时）：
```
任一会话「要你介入」  → 🔴 红（最高优先级）
否则任一「出错」      → ⚡ 红快闪
否则任一「干活中」    → 🔵 蓝呼吸
否则有「刚完成」      → 🟢 绿（短暂，几秒后转空闲）
否则全部空闲         → ⚫ 暗
```

**运行方式**：开机自启（macOS launchd LaunchAgent），后台常驻，无需用户干预。

### 3.3 显示层（ESP32 固件）

傻瓜显示端。职责：
1. **联网**：开机读 NVS 里的 WiFi 凭据自动连接；连不上则进入配网模式（见 §6）。
2. **接收指令**：跑一个小 HTTP 服务，接收转译器推来的显示指令。
3. **渲染**：根据指令驱动 WS2812 出对应颜色 + 动效（呼吸、慢闪、心跳脉冲、渐暗、扫描、分段）。
4. **看门狗**：超过 ~30 秒没收到任何消息 → 切"失联"显示。
5. **失联恢复**：恢复收到推送后立即按指令回到正确显示。

**固件不做**：不解析 agent 事件、不存业务状态、不做合并逻辑。它只认"显示指令"这一种输入。

## 4. 状态模型（颜色 + 动效）

| 状态 | 显示 | 触发 |
|---|---|---|
| ⚫ 空闲 | 暗/灭 | SessionEnd / 无活动 |
| 🔵 干活中 | 蓝**呼吸**，按工具分色：写码=蓝·跑命令=紫·搜索=青 | UserPromptSubmit / PreToolUse |
| 💓 推进一步 | 每次工具调用一个**心跳脉冲** | PreToolUse |
| 🟢 完成 | 绿亮 3-5 秒 → **渐暗**回空闲 | Stop / agent-turn-complete |
| 🔴 要你介入 | 红**慢闪** | Notification / approval-requested |
| ⚡ 出错 | 红**快闪一下** → 弹回干活色 | PostToolUse 报错 |
| 🟠 失联 | 暗琥珀/白**慢呼吸**（明显区别于以上各色） | 看门狗 ~30s 没收到消息 |
| 🚀 开机 | 上电跑个扫描动画 | SessionStart / 上电 |

**铁律**：「空闲(暗)」和「失联(琥珀)」必须严格区分——一盏卡死在蓝色的灯比没有灯还坏。

## 5. 多会话显示

- **单会话**：整圈灯做合并的丰富动效（呼吸/心跳/渐暗）。
- **多会话**（如 Claude Code + Codex 同开）：灯环按段分配，每个会话占一段，各显各的状态。
- 转译器每次推送「活动会话列表 + 各自状态」，ESP32 据此决定整圈渲染还是分段渲染。

## 6. 通信与配网

### 6.1 状态推送（WiFi + HTTP）
- 转译器 → ESP32 主推送路径。一次性命令友好，电脑端零额外软件负担。
- **API（初步）**：
  - `POST /state`：body 为 JSON，含活动会话列表及各自 `{agent, session_id, state, tool}`。设置显示并重置看门狗。即使状态没变，转译器也每 ~5-10 秒重发一次作为心跳。
  - `GET /health`：返回固件当前显示状态/存活信息，供转译器在重连时校验与重新同步。
- **看门狗**：固件 ~30s 内没收到 `/state` → 进入失联显示。

### 6.2 配网（v1：自建热点 + 网页）
固件刷入后不知道家里 WiFi 密码，且密码不应硬编码（否则换网要重烧）。配网方案：
- **v1：WiFiManager 网页配网**——首次开机/连不上时，ESP32 自建临时热点（如 `VibeLamp-Setup`）；手机/电脑连上后自动弹出配网网页，选 WiFi 填密码；存入 NVS、重启、连上。零 App，任何浏览器可用，久经考验。连不上时自动重开配网门户。
- **v1.1 备选：BLE 配网**——直接套用乐鑫官方 ESP BLE Provisioning App，不用自己写 App。

### 6.3 BLE（v1.1）
- WiFi 断时转译器自动切 BLE 继续推送，双通道冗余，灯几乎不会真失联。
- v1 先把 WiFi 主链路 + 失联处理做扎实，BLE 兜底后补，避免一上来双通道把固件搞复杂。

## 7. 固件生命周期（澄清"烧录 vs 运行时"）

| 阶段 | 做法 | 频率 |
|---|---|---|
| ① 烧录固件 | USB 插一次刷入 flash（v1.1 起支持 OTA 无线升级） | 一次（改固件才再刷） |
| ② 首次配网 | 运行时把 WiFi 凭据存入 NVS | 一次（断电不丢） |
| ③ 日常运行 | 上电自动连 WiFi，等推送；状态变化都是运行时网络数据 | 永不插线 |
| ④ 换网/搬家 | 连不上自动重开配网，重配一次 | 偶尔 |

状态变化（蓝/绿/红）是**运行时 HTTP 推送的网络数据**，与烧录无关，绝不需要每次重刷。

## 8. 断线与健康处理（核心可靠性）

| 场景 | 表现 |
|---|---|
| ESP32 掉 WiFi | 看门狗超时 → 🟠 失联显示 |
| 转译器进程挂了/没启动 | 收不到心跳 → 🟠 失联显示 |
| 转译器推送时灯不在/不可达 | 转译器静默重试 + 缓存最后状态，**不崩** |
| ESP32 断电重启 | 开机动画 → 等转译器重推当前真实状态 |
| 某会话异常退出（漏发结束事件） | 转译器超时兜底回空闲，避免某段永远卡蓝 |
| Mac 睡眠 / 无会话 | 正常 ⚫ 空闲——与 🟠 失联严格区分 |

机制：**心跳 + 看门狗 + 恢复后重新同步**。

## 9. 硬件

| 部件 | v1（打样） | 成品（可选升级） |
|---|---|---|
| 主控 | 手上的 ESP32 | ESP32-C3 SuperMini / XIAO ESP32-C3（更小更便宜，功能零差别） |
| 显示 | 面包板 + 分立 LED（红/绿/蓝接 3 个 GPIO）先跑通链路 | WS2812 灯环（16 灯） |
| 扩散 | 无（裸灯即可） | 乳白亚克力扩散罩 / 白色 PLA 3D 打印外壳（淘宝搜「乳白亚克力圆环」「LED 扩散板 乳白」，非「磨砂导光罩」） |
| 供电 | USB | USB |

- WS2812 由 ESP32 的 RMT 硬件外设驱动（FastLED/NeoPixelBus 库），时序稳，不受 WiFi 干扰。
- WiFi + BLE 共存对低流量场景无压力。
- 成本不是约束，选料按"最顺手好看"来。

## 10. 测试策略

- **转译器**：单元测试状态合并逻辑（各种会话组合 → 期望显示）、超时兜底、推送失败重试。可用假的 ESP32 端点（mock HTTP server）。
- **固件**：可先用串口日志验证收到的指令与渲染分支；看门狗超时用断网模拟。
- **端到端**：打样阶段用分立 LED 跑通「真实 Claude Code 会话 → 灯变色」整条链路再上 WS2812。
- **断线**：手动拔网/杀进程/断电，验证失联显示与恢复重同步。
